Resource Monitor is a Windows desktop application that displays system hardware information such as CPU load, CPU frequency, memory load, and disk load in a transparent, HUD-like, always-on-top overlay. This allows hardware to be monitored without the use of a separate window such as Task Manager's performance tab.

The overlay can be toggled on and off through a hotkey and is customizable. Individual counters can be moved around and their font, font color, font size, and background can be changed. Resource Monitor can also be launched on Windows startup and will preserve its state across uses.

Win32 Sadness

The first step in learning anything new consists of writing a simple 'hello world' program. These usually take up a few lines, are easy to understand, and provide a great starting point. Charles Petzold, author of Programming Windows, writes: "The original hello world program in the Windows 1.0 SDK was a bit of a scandal. HELLO.C was about 150 lines long, and the HELLO.RC resource script had another 20 or so more lines... Veteran programmers often curled up in horror or laughter when encountering the Windows hello-world program."

So the Win API learning curve is quite steep, but developers are rewarded with access to low-level operations that allow their applications to have more flexibility and power. Back when Win API was first being developed, Microsoft had to support various CPU architectures and compilers. There were differences in 16 bit and 32 bit processors. There were differences in pre-ANSI C and unstandardized C++ compilers. There were differences in Intel, PowerPC, and ARM chips. Microsoft had to move from single-core to multi-core systems, deal with segmented memory, and support international encoding before Unicode existed. All of this results in many, Win32-specific custom types that seek to standardize development.

For example, a data type of name BOOL exists. This is not to be confused with the traditional C++ bool, which is a boolean variable that takes either true or false. Rather BOOL is an int that takes or returns values TRUE and FALSE which are not booleans but ints as well. This results in functions as such as BOOL CALLBACK EditDlgProc(HWND hwndDlg, UINT message, WPARAM wParam, LPARAM lParam); which at first glance seems to return a true or false depending on whether or not it succeeds but in actuality returns the int 1, 0, or a numeric error code. If you're like me and wondering why this isn't written as int CALLBACK EditDlgProc(HWND hwndDlg, UINT message, WPARAM wParam, LPARAM lParam);, it's because it was once considered good practice to use abstract types rather than primitive types such as int, bool, and char.

Here's another example. Word. I know what a word is. You know what a word is. A word is a collection of letters. So when Win32 documentation lists something like DWORD, I was absolutely floored to find out that DWORD is actually an unsigned long int. So instead of passing in a word into DWORD, I would be passing in numbers. Unsurprisingly, WORD is an unsigned short int. This leads to functions like
HFONT CreateFontA(
int cHeight,
int cWidth,
int cEscapement,
int cOrientation,
int cWeight,
DWORD bItalic,
DWORD bUnderline,
DWORD bStrikeOut,
DWORD iCharSet,
DWORD iOutPrecision,
DWORD iClipPrecision,
DWORD iQuality,
DWORD iPitchAndFamily,
LPCSTR pszFaceName
);
where the unassuming programmer might think the parameter DWORD bItalic would take some type of word that describes the italic setting. Maybe something like 'italic' or 'normal'. You would actually pass in TRUE which represents the numeric value 1 and can be stored as an int. Confusingly, there are also regular int parameters that CreateFontA takes along with its DWORD parameters that are not words, but ints.
